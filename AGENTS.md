# AGENTS.md

本文件是本仓库的工程协作约束。所有自动化代理和人工维护者在修改本项目时都应遵守。

## 总体原则

- 以交付可运行、可验证、可维护的本地 Codex 会话管理工具为主线。
- 保持实现直接、清晰、低冗余，不为暂时不存在的场景提前堆叠复杂设计。
- 优先使用 Python 标准库和仓库已有脚本；新增依赖前必须确认它能显著降低复杂度或风险。
- 修改功能时同步检查关联路径，及时收敛重复实现、失效分支和调试残留。

## 项目边界

- `app.py` 只负责命令行入口和调度。
- `codex_store/sessions.py` 只负责会话读取、筛选、分页和导出。
- `codex_store/backups.py` 只负责备份扫描、可恢复性判断和 rollout 备份。
- `codex_store/sqlite_utils.py` 只负责 `threads` 表修复和 SQLite 写入辅助。
- `codex_store/jsonl_utils.py` 只负责 jsonl 历史与索引辅助操作。
- `codex_store/mutations.py` 只负责重命名、删除、彻底删除和恢复。
- `web/server.py` 只负责本地 HTTP 请求处理、重定向和错误返回。
- `web/actions.py` 只负责打开终端、批量恢复会话等桌面系统动作。
- `web/ui.py` 只负责 HTML 结构渲染和页面数据拼接。
- `web/styles.py` 只负责 CSS 聚合入口。
- `web/style_sections/` 只按页面区域维护 CSS 分区，不放业务逻辑。
- `web/scripts.py` 只负责浏览器端脚本渲染。
- `tools/` 只放安装、启动和本地运行脚本。
- `macos/` 和 `linux/` 只放对应平台的桌面启动模板和资源。

新增文件前先判断是否真的需要独立模块；如果能放在现有职责边界内完成，不随意拆文件。

## 实现要求

- 优先选择可读、稳定、容易排查的实现。
- 避免过深封装、过早抽象和无收益的设计模式。
- 涉及 Codex 底层数据的写操作必须保持可回退：写入前创建备份，失败时返回清晰错误。
- 涉及 SQLite、jsonl、rollout 文件的逻辑必须考虑缺文件、坏行、重复 ID、损坏备份和权限失败。
- 不静默吞掉会影响用户判断的写入错误；页面应能看到明确失败原因。

## 前端结构与样式约束

- 当前前端采用零构建渲染方式，由 `web/ui.py`、`web/styles.py`、`web/style_sections/`、`web/scripts.py` 分别维护 HTML、CSS 聚合、CSS 分区和 JS。
- CSS 结构要尽量与 DOM 结构对齐，按界面区域组织：
  - 页面基础与公共控件
  - 页头区域
  - 搜索区域
  - 列表与表格区域
  - 分页与批量操作区域
  - 弹窗与快捷命令区域
  - 主题覆盖区域
  - 响应式区域
- 一个明确界面区域对应一段样式，不通过零散补丁让样式跨区叠加。
- 多个区域复用的按钮、输入框、表格、图标按钮等样式应放在公共控件区。
- 主题样式只覆盖颜色、阴影、背景、边框等视觉变量，不应改变核心 DOM 结构和交互逻辑。
- 修改前端代码时必须重复检查：
  - CSS 和 DOM 是否仍然对齐
  - 重复样式是否可以合并
  - 公共样式和组件私有样式边界是否清晰
  - 表格列名和内容是否仍然对齐
  - 弹层、提示、复制反馈、标题编辑等交互是否不改变行高和布局
- 前端代码应持续朝结构清晰、直接、易维护的方向收敛，不允许通过不断追加补丁让样式和结构越来越混乱。

## 注释要求

- 代码注释统一使用中文。
- 以下场景必须写中文注释：
  - 模块用途
  - 函数职责
  - 关键功能流程
  - 特殊处理、边界条件、兼容性原因
  - 容易误解的数据库、脚本或运维操作
- CSS 应按组件或公共样式块添加中文注释，说明该段样式服务的界面区域或复用目的。
- 注释要解释“为什么这样做”，避免无信息量注释。

## 测试与验证

当前仓库暂未引入正式测试框架。新增核心逻辑时，应优先补可自动运行的测试；在测试框架建立前，至少执行以下检查：

```bash
python3 -m py_compile app.py config.py models.py codex_store/*.py web/*.py web/style_sections/*.py
bash -n tools/run_local_ui.sh
bash -n tools/install_mac_app.sh
bash -n tools/install_linux_app.sh
git diff --check
```

修改以下功能时，需要做对应冒烟验证：

- 标题修改：临时改一条现存会话标题，再改回原值。
- 删除：确认现存视图消失，并出现在已删除或全部会话视图中。
- 彻底删除：确认已删除和全部会话视图都不再出现该记录。
- 恢复：确认可恢复记录恢复后出现在可继续视图中。
- 启动脚本：确认已有实例复用和首次启动都能打开页面。

后续如果引入测试框架，核心逻辑测试覆盖率目标为 70% 以上；测试应验证真实行为，不为覆盖率堆无意义用例。

## 清理要求

- 提交前清理缓存、临时文件和构建残留。
- 不提交 `.DS_Store`、`__pycache__`、日志、临时 `.app`、`dist/` 等本地产物。
- 每次推送前固定执行：
  - 查看 `git status --short`
  - 清理缓存和临时文件
  - 运行语法检查和脚本检查
  - 运行必要的功能冒烟
  - 检查 diff 是否只包含本次目标改动
